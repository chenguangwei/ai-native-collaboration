# 🔌 API 开发规范

> 作用域: `src/api/**`, `src/handlers/**`, `src/server/**` 及所有后端接口层。
> **注意**：这是可选阅读的业务域知识 ROM，供 AI 在构建后端功能时索引，不再强制霸占热内存系统提示词资源。

## 1. 响应标准格式

所有 API 端点建议遵守统一的数据返回格式：

### 成功响应
```json
{
  "code": 0,
  "data": { ...业务数据... },
  "message": "success",
  "timestamp": 1699999999999
}
```

### 错误响应
```json
{
  "code": 400,
  "message": "Invalid email format",
  "details": { ...进一步的结构化错因... }
}
```
*禁止直接返回原生底层错误信息（如数据库查错语句、堆栈轨迹等）防止安全泄露。*

## 2. 请求处理原则

- **输入必验证**: 永远不信任外部输入，必须在系统边界层（Controller/Handler）使用框架对应的 Schema 校验机制。快速失败（Fail-fast）。
- **异常全捕获**: 所有的处理函数必须包裹在全局异常处理或 Try-Catch/Try-Except 中。
- **日志全覆盖**: 所有的入口请求必须有基础请求日志，关键节点（创建、支付、删除等）要有行为审查日志。
- **状态码合规**: 遵守 HTTP 语义（400: 参数错, 401: 未认证, 403: 权限错, 404: 找不到, 500: 服务器内部错误）。

## 3. 命名与路由设计

- **Restful 倾向**: 推荐基于资源（Resources）命名，使用名词且为复数（如 `/api/v1/users`），禁止动词入路（如 `POST /api/v1/createUser`）。
- **版本控制**: 新接口必须包含版本控制（`/v1/`），不可直接挂载在 `/api` 根目录。
- **方法适配**: 正确使用 HTTP 方法：
  - GET: 获取资源（无副作用）
  - POST: 创建资源
  - PUT: 完全覆盖或替换资源
  - PATCH: 部分更新资源
  - DELETE: 删除资源
