# 🤔 RAM 内存架构 FAQ (常见问题解答)

在我们将原本依赖 JSON 配置、按角色严格硬隔离的多层小文件，激进地合并成了仅仅 3 个共享 Markdown 文件后，很多从传统协作模式迁移过来的开发者可能会有以下疑问。

---

### Q1: 这几个公共文件多角色或多人并发运行时，会有写入冲突吗？

**不会产生结构性灾难，由于 Git 存在，这是受控的、反而是更优的协作模式。**

*   **旧架构的“防并发幻觉”**：以前依赖 bash 脚本加锁（`lock.sh`）去隔离不同角色，看似规避了冲突，但这反而导致信息互不流通，前端不知道后端改了哪（信息孤岛）。
*   **新架构利用 Git 完美合并**：现在这些文件变成了全队的“公共工作白板”。因为我们的工程底层依然使用 Git 进行版本控制，如果产品经理勾选了 `active-task.md` 的前几项，后端工程师勾选了后几项，推送到远端 Git 仓库时，Git 原生的 **自动合并 (Auto-merge)** 会极为完美地处理这些不同行的文本追加。
*   **遇到极少数同行冲突时**：手动把冲突化解（解决两个 Markdown 进度大纲的重叠）也远比排查一个卡住死锁的晦涩“防并发脚本”要直观、安全得多。

### Q2: 在一个大本子里，如何分清哪个是“自己角色”的内容？

**摒弃物理隔离，改用“大纲语义分区”与强制的“落款签名”。**

*   **在 `active-task.md` 找活干（看区块标签）**：由于新架构极力推崇基于大模型的智能分析。任务大纲是以 **Feature（功能特性）** 为单位去规划的：
    ```markdown
    ## 模块：用户登录
    - [x] [产品] 确定无需手机号强制绑定的 PRD
    - [ ] [后端] 设计 Oauth2 登录库及接口
    ```
    当你或 AI 以“后端”的身份介入时，AI 会一眼看懂：带有 `[产品]` 标签并且被标记为 `[x]` 的行是确定过的定稿，自己只需要去推进处理 `[后端]` 留空的 Checkbox。
*   **在 `handoff.md` 取证、追责（看落款）**：AI 底层的指令规范（`02-memory-protocol.md`）已经被我们硬性挂载了“强制署名”的逻辑要求。每次大模型写入交接单时，都会默认打上如 `**[前端工程师 - 2026-03-28 15:30]**:` 格式的前置落款。各个工种遗留的话术和遗留的 Blocker 边界异常清晰。

### Q3: AI 是如何“自动”完成记忆文件更新的？

**它是依靠大模型底层的拦截级别 System Prompt (思想钢印) 隐式驱动的，没有用笨拙的后台脚本。**

由于脚手架会在项目环境下永远静默向 AI 挂载 `.claude/rules/02-memory-protocol.md` 规则。这如同一道不可违背的思想钢印，给 AI 设定了极高的优先级准则：

*   **“当达成开发里程碑进度，或遇到严重死胡同报错时，在最终回答用户之前，你必须先使用工具更新那些大纲记忆文件！”**
*   于是，当你在终端中喊 AI 去跑满全套测试代码，测试全部绿灯。它原本正准备直接向你脱口而出：“老板我搞定了”。但这道钢印在后台拦住了它，于是模型会自动启动后台的内部计算，强行调用 `replace_file_content` 工具，悄悄把 `active-task.md` 里对应的待测复选框打个钩 `[x]`。一切落盘后，这才向终端喷吐胜利的结果。整个“验收/存档/总结”流程变成了极致顺滑的原生 LLM Agent 自驱。
