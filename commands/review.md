# /review

> 唤醒【审查员】：执行 PR 级的 Code Review，拦截漏洞

## 触发条件

- 代码提交前
- 收到 review 请求
- 代码合并前

## 执行流程

### 1. 代码扫描

#### 1.1 静态分析

```bash
# 运行 ESLint
npm run lint

# 运行 TypeScript 检查
npm run type-check

# 运行安全扫描
npm audit
```

#### 1.2 依赖检查

- 检查新依赖是否必要
- 检查依赖版本是否安全
- 检查是否有未使用的依赖

### 2. 代码审查清单

#### 2.1 代码规范

- [ ] 遵循 ESLint 规则
- [ ] 命名清晰有意义
- [ ] 没有硬编码值
- [ ] 没有 console.log

#### 2.2 逻辑正确

- [ ] 边界条件已处理
- [ ] 错误处理完善
- [ ] 没有空指针风险
- [ ] 并发安全

#### 2.3 性能

- [ ] 没有不必要的重渲染
- [ ] 避免内存泄漏
- [ ] 数据库查询已优化

#### 2.4 安全

- [ ] 输入已验证
- [ ] 没有 SQL 注入
- [ ] 没有 XSS 漏洞
- [ ] 敏感信息未泄露

#### 2.5 测试

- [ ] 单元测试覆盖
- [ ] 关键路径有测试
- [ ] 测试可读性强

### 3. 审查输出

```markdown
## Code Review 报告

### 整体评价
[优秀/良好/需要修改]

### 必须修改 (Blocking)
1. [问题描述] - 文件:行号
   - 建议修复方式

### 建议修改 (Non-blocking)
1. [改进建议]

### 赞赏点
- [做得好的地方]
```

---

## 审查标准

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| Blocking | 必须修改 | 禁止合并 |
| Non-blocking | 建议修改 | 可选合并 |
| Nitpick | 小问题 | 可忽略 |

---

## 后续行动

1. 修复 Blocking 问题
2. 重新提交
3. 调用 `receiving-code-review` 处理反馈
