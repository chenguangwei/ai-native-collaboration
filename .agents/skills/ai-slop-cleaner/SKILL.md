# /ai-slop-cleaner

系统化清除 AI 生成的代码味道，不改变行为。

## 使用场景

**使用**：
- 用户说"deslop"、"反AI味"、"代码太臃肿"、"清理一下"
- 实现完成后有重复逻辑、死代码、无意义封装层
- 代码能跑但感觉"很 AI"

**不使用**：
- 需要新功能的情况（这是清理，不是开发）
- 需要整体重设计（不是增量清理）

## 执行原则

- **保留行为**：除非用户明确要求，否则不改变功能
- **先测试后改**：有测试的代码才能安全清理
- **删除优先于添加**：能删则删，不能删才改
- **范围受控**：只清理指定文件，不扩散到无关代码

## 执行步骤

### Step 1: 扫描并识别 Slop

读取目标文件，识别以下气味：

| Slop 类型 | 表现 | 严重度 |
|----------|------|--------|
| 无意义注释 | `// 将 count 加 1` / `// 返回用户` | 低 |
| 过度封装 | 单次使用的 helper 函数 / 一行的 wrapper | 低 |
| 重复逻辑 | 3+ 处相同代码片段 | 中 |
| 死代码 | 未使用的变量/函数/导入 | 中 |
| 防御性过度 | 永远不会发生的 null 检查 | 低 |
| 过度抽象 | 为1个用例设计的接口/工厂/策略模式 | 高 |
| 模板化命名 | `handleClick`/`processData`/`doSomething` | 中 |

### Step 2: 制定清理计划

在执行任何改动前，输出清理计划：
```
## 清理计划

**目标文件**：{文件路径}
**发现的问题**：{n} 个

| 问题 | 位置 | 类型 | 操作 |
|------|------|------|------|
| 无意义注释"将count加1" | L42 | 无意义注释 | 删除 |
| `formatUserName` wrapper | L89-91 | 过度封装 | 内联 |
| ... | | | |

**行为保护**：
- [ ] 是否有测试覆盖？{是/否}
- [ ] 是否需要先补测试？{是/否}

继续执行清理计划？
```

### Step 3: 行为锁定（如果有测试）

运行现有测试，确认基线：
```bash
# 根据项目技术栈选择
npm test / mvn test / pytest
```

如果没有测试且改动风险较高：告知用户建议先补测试。

### Step 4: 执行清理

按计划逐项执行，使用 Edit 工具：
- 删除无意义注释
- 内联单次使用的 wrapper
- 合并重复逻辑
- 删除死代码
- 修复模板化命名

**每次改动后都要检查**：逻辑是否与改动前等价？

### Step 5: 验证

```bash
# 重新运行测试
npm test / mvn test / pytest

# 编译检查
npm run build / mvn compile
```

### Step 6: 报告

输出清理摘要：
```
## 清理完成

**处理文件**：{文件列表}
**删除行数**：{n} 行
**修改项**：
- 删除了 {n} 处无意义注释
- 内联了 {n} 个单次使用的 wrapper
- 合并了 {n} 处重复逻辑
- 删除了 {n} 个死代码

**测试状态**：{通过/失败}

建议提交：git add {文件} && git commit -m "refactor: deslop {文件名}"
```
