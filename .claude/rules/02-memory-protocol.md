# 🔄 记忆压缩交接与人格锁死法案 (Memory Protocol & Persona)

> 为了防止长上下文衰减导致大模型退化为“老好人通才”或形成满屏废话的“流水账幻觉”。

## 🔌 实时人格约束锚点 (Persona Enforcement)
当你准备就重大决策或长篇代码发表回复时：
1. **自动探测**：如果你发现系统中存在物理落盘文件 `memory/.current_persona`，必须优先强制应用该文件中所定义的**偏激立场和严苛要求**。
2. **重载覆盖**：由 `memory/.current_persona` 加载出来的极限定律，拥有最高优先级，它将立刻暂时覆盖你在规则中被默认教导的通用性格。在会话中偏离这套物理锁死的人格是绝对不允许的。

## 📍 记忆表更新触发条件 (Must Trigger)
发生以下任一情况，你必须打断手中普通代码，向 `memory/` 执行物理落盘同步：
1. 完成了重大阶段进度点 (Milestone)。
2. 遇到被严重 Block 且需要人类介入的死胡同。
3. 切换上下文接手一个无关的新需求。

## 💾 记忆压缩机制 (Sliding Window Memory)
位于 `memory/` 的核心工作日志必须扁平且精简：
### 1. 📋 `active-task.md` (主线阶段待办)
- 用 `[ ]` 和 `[x]` 管理拆解的宏观树状目标。绝不记流水账废话。
### 2. 🤝 `handoff.md` (会话微观交接单)
- **强制滑动窗口压缩 (Sliding Window Compression)**：一旦该文档接近 50 行，必须立刻使用 `replace_file_content` 将过往的老旧流水账无情删除，并用全局视角的总结合并代替（绝不允许无脑 Append 追加）。
- 每次写必须包含：前置署名落款（如 `**[前端架构 - 2026-03]**`）、刚才改了啥、现在的 Blocker 是啥、下一步该谁干什么。
### 3. 🧠 `project-facts.md` (避坑准则)
- 如果某个 Bug 是这个项目隐藏的“暗黑脾气”，请提炼为一句金科玉律强力写入此地。
