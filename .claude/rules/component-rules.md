# 🎨 UI 组件规范

> 作用域: `src/components/**`, `src/ui/**`, `src/views/**` 相关的组件结构边界。

## 1. 组件结构分离

遵循“单一职责原则 (SRP)”，避免“面条型结构”：

- **展示与状态分离**: UI 组件层只负责渲染（纯组件），复杂的业务逻辑必须抽离到对应的 Hook（或逻辑控制层）。
- **文件拆分**: 若组件膨胀至 300 行以上，强制拆分出类型定义文件（`Types`）、样式定义文件（`Styles`）、工具函数（`Utils`）子模块。不准让一个文件挤满类型、状态、API 获取和繁杂的 DOM 结构。
- **扁平导入**: 推荐使用基于 `index` 的桶导入（Barrel export），对外暴露统一接口。

## 2. 强类型接口边界

无论使用哪种语言和框架（TypeScript/Props/PropTypes），都必须为组件建立严格的契约边界：

- **明确声明**: 每一个组件的输入属性（Props）必须拥有强类型定义。
- **禁绝 Any**: 绝对禁止使用 `any` 绕过类型系统。
- **合理默认值**: 必须为可选参数及布尔型控制标志设定安全的默认参数。
- **文档说明**: 将 Props 当作对外发布的库 API，为每个属性编写标准注释（特别是那些复杂数据结构）。

## 3. 样式管理准则

- **杜绝内联魔法**: 一切结构化样式必须抽离至专门的样式层（CSS Modules, Tailwind Classes, Styled Components），禁止散落的硬编码内联样式（`style={{ color: 'red' }}`）。
- **可复用常量**: 复用的变体与配置（Variants、Design Tokens）应当作为业务规则抽离，而不是用条件语句层层嵌套去拼凑类名。

## 4. 性能最佳化与行为模式

- **避免闭包陷阱**: 所有的事件绑定处理器，原则上都必须通过防抖或记忆化包裹，避免每次组件渲染时重复创建闭包挂载 DOM 事件。
- **减少副作用爆炸**: 生命周期钩子（或 `useEffect`）里，每一项外部资源的订阅必须匹配同等级别的卸载/销毁，杜绝内存泄漏。
