# 🎯 技能触发机制

> 依据上下文自然语言自动匹配并触发最佳实践。不需要死记正则表达式，掌握意图核心即可。

## Level 1: 强制触发 (Must trigger)
在面对以下核心技术场景时，**无须人类确认**，直接加载调用对应的能力与策略：

- **重大构建/类型检查失败**: 直接进入 Debug/Resolution 轨道，缩小排查范围（最小 diff 修复原理）。
- **任务目标模糊不清**: 严禁擅作主张动代码。强制启动需求访谈/反问澄清环节，将不确定性降低到 20% 以下并获得计划再开工。
- **刚收到人类或机器的 Code Review 反馈**: 不盲目同意。先验证反馈的合理性和技术可行性后再逐点修改，确保不引入新的问题。
- **完成任何重大逻辑实现前**: 强制调用 `verification-before-completion`（比如运行测试套件或代码内检查脚本），绝大部分情况必须在代码可以 run 跑通的基础上方可声明“已完成”。

## Level 2: 推荐触发 (Highly Recommended)
遇到长线或探索属性的任务，主动向用户推荐接入结构化思维或执行专项命令/工具：

- **全新的复杂功能建设**: 推荐生成详尽的多步实现规划（Plan）。
- **架构重构与痛点修复**: 遇到复杂或老旧代码的大规模改写，呼起 `using-git-worktrees` （或其他隔离分支技术）保障工程不被毁坏。
- **编写极度关注容错的功能**: 启动测试驱动开发 `test-driven-development`。
- **PR 提交或分支合并前**: 最好进行一套统一的代码格式化、代码异味扫描（Code Smell Check）。

## Level 3: 可选辅助 (Optional)
根据实际的上下文（例如是否处于纯后端 Node环境或纯前端设计阶段）再定夺：

- 跨平台的 UI 组件重设（使用相关 UI 最佳实践或 `frontend-design` 技巧）。
- 重大基建工具的设计决策（建议进行双边 AI 核查或二次求证）。

## 核心要点 / Meta Guideline

绝不要为了触发技能而硬凹特定的 Prompt。所有的技能核心都应该基于 **“做计划 -> 隔离环境验证 -> 严谨测试并修复 -> 反思与确认”** 这一套飞轮逻辑进行。遇到棘手 Bug 第一时间不要猜代码改哪行，而是梳理执行链路（`systematic-debugging` 核心逻辑）。
