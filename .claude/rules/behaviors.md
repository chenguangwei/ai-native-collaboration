# 📜 核心行为准则

> AI 的肌肉记忆 - 全局始终加载

## 1. TDD 先行 (Test-Driven Development)

### 原则
- **先写测试，再写实现**
- 测试失败是开发的起点，不是终点
- 测试覆盖率未达标禁止提交

### 操作流程
1. 理解需求后，先写测试用例
2. 运行测试，确认失败
3. 编写最简代码让测试通过
4. 重构代码，保持测试通过

### 禁止事项
- ❌ 禁止先写实现再补测试
- ❌ 禁止跳过测试直接提交
- ❌ 禁止提交失败的测试

---

## 2. 代码原子性 (Atomic Commits)

### 原则
- **每次提交只做一件事**
- 提交信息必须清晰描述改动
- 保持提交历史可追溯

### 操作流程
1. 完成一个独立改动后立即提交
2. 使用清晰的提交信息
3. 确保每次提交都能独立运行

### 提交信息格式
```
<类型>: <简短描述>

<详细说明（可选）>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### 类型前缀
- `feat`: 新功能
- `fix`: Bug 修复
- `refactor`: 重构
- `docs`: 文档
- `test`: 测试
- `chore`: 维护

---

## 3. 人类确认原则

### 原则
- **严禁未经确认直接 push**
- 重大改动需要人类审批
- 保持透明，不做越权操作

### 需要确认的场景
- 推送到远程分支
- 删除或覆盖文件
- 修改配置文件
- 引入新依赖
- 发布版本

### 操作流程
1. 完成代码后展示改动
2. 等待人类确认
3. 执行 push/release 操作

---

## 4. 安全与隐私

### 原则
- 不提交敏感信息
- 不泄露 API 密钥
- 遵守许可证

### 敏感信息清单
```
- API keys
- 数据库连接字符串
- 用户隐私数据
- 内部凭证
```

### 沟通原则

## 6. 不可变性原则 (Immutability — CRITICAL)

### 原则
- **永远创建新对象，禁止原地修改现有对象**
- 不可变数据防止隐藏副作用，使调试更容易，并支持安全并发

### 示例
```
// 错误做法:
modify(original, field, value)  // 原地修改 original

// 正确做法:
update(original, field, value)  // 返回含修改的新副本
```

### 适用范围
- JavaScript/TypeScript: 使用 spread (`{...obj}`, `[...arr]`)，禁止 `push`/`splice`/直接赋值属性
- Java: 使用不可变 DTO，Record 类型优先
- Python: 使用 dataclass(frozen=True) 或返回新实例

## 7. 代码质量标准 (Code Quality)

### 文件组织
- **200-400 行**为宜，**最大 800 行**
- 按特性/领域组织，而非按类型
- 高内聚，低耦合；从大型模块中提取工具函数

### 函数规范
- 函数体 **<50 行**
- 嵌套深度 **≤4 层**
- 无硬编码值（使用常量或配置）

### 错误处理
- 每层都要显式处理错误
- UI 层提供用户友好的错误信息
- 服务端记录详细错误上下文
- **禁止静默吞掉错误**

### 输入验证
- 在系统边界处验证所有输入（用户输入、API 响应、文件内容）
- 使用 schema-based 校验（Zod / Pydantic / Bean Validation）
- 快速失败并给出清晰的错误信息
- **永远不信任外部数据**

### 提交前自查清单
- [ ] 代码可读，命名清晰
- [ ] 函数 <50 行，无深嵌套
- [ ] 文件 <800 行，按特性组织
- [ ] 每层都有错误处理
- [ ] 系统边界有输入验证
- [ ] 使用不可变模式，无原地修改
- [ ] 无硬编码值

## 8. Agent 编排规范 (Agent Orchestration)

### 并行原则
对于独立操作，**始终**使用并行 Task 执行：

```
✅ 好的做法：并行执行独立任务
- Agent 1: 安全性分析（auth 模块）
- Agent 2: 性能检查（cache 系统）
- Agent 3: 类型检查（工具函数）

❌ 不好的做法：无依赖时仍然串行执行
```

### 即时 Agent 调用（无需用户提示）
| 场景 | 使用 Agent |
|------|-----------|
| 复杂功能请求 | `planner` |
| 刚写完/修改代码 | `code-reviewer` / `frontend-reviewer` / `java-reviewer` |
| Bug 修复或新功能 | `tdd-guide` |
| 架构决策 | `architect` (使用 `/plan-architect` 命令) |
| 安全敏感代码 | `security-auditor` |

### 多视角分析
对于复杂问题，使用分角色子 agent：
- 事实核查者
- 高级工程师视角
- 安全专家视角
- 一致性审查
