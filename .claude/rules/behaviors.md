# 📜 AI 核心行为与指令底座

> 全局始终加载的最底层 AI 宪章，规范执行流走向。

## 1. 绝对行为底线 (Critical Principles)

1. **不可变性原则 (Immutability)**：永远创建新对象，绝对禁止原地修改现有对象！JavaScript/TypeScript 大量采用 spread (`{...obj}`) 等语法；在 Python 中应用 `dataclass(frozen=True)`。
2. **人类确认机制 (Confirmation Gate)**：在执行核心高危的 Git Push / Delete / Overwrite 操作、修改配置、引入第三方依赖时，**必须**挂起任务显式征求人类指令同意。禁止“越权擅自操作”。
3. **安全先行 (Security First)**：在读取或输出任何涉及 API Keys / 密码口令字符串、涉密日志、敏感用户信息的地方，自动拦截并停止在公屏抛出。

## 2. 工作流范式 (Workflow Paradigms)

### TDD 驱动 (Test-Driven)
**“先写测试，再写实现”**：遇到任何功能，无论你多自信，都应首先产出对应模块的 Failing Tests。测试通过是你能够宣告任务搞定的唯一凭证！不允许提交掩耳盗铃式的通过率测试（例如全写 `expect(true).toBe(true)`）。

### 原子化微提交 (Atomic Commits)
**“每次变更只集中一件事”**：提交历史要高度可追踪、独立化。不准把样式调整与破坏性的 API 重构混战在一个 Commit 里提交。每个 Commit Message 中请自带“决策解释”。

## 3. 架构与代码设计操守 (Code Quality)
- **拒绝“巨石”**：单个核心逻辑文件绝不应超过 500 行。
- **浅层嵌套**：函数的圈复杂度、缩进级别需严格限制在 3 到 4 层（杜绝箭头函数的 V 型恐怖深渊），提前 `return`（Guard Clauses）。
- **零信任输入**：系统边界的输入数据永远不可被信任，所有暴露给外界的方法必须具备严格的参数前置推论验证。
- **消灭静默异常**：所有的 `Catch` 或底层错误处理不仅要显式记录，还得保证上层能收到恰当反馈（绝不再把错误吃掉吞吐无声）。

## 4. 模型脑力与能力分发
在设计方案和自我排查时，根据面临的问题选用或推荐正确的脑力轨道：
- **微型与机械问题**：用 Haiku / 轻便模型 快速查找、分析文件目录结构。
- **中型功能或重构**：用 Sonnet / 标准模型 生成功能核心逻辑实现。
- **深度并发死锁或系统性技术选型**：开启 Opus / 深度推理模型 处理全局宏观系统架构或高并发安全性痛点。
