# 📜 核心行为准则

> AI 的肌肉记忆 - 全局始终加载

## 1. TDD 先行 (Test-Driven Development)

### 原则
- **先写测试，再写实现**
- 测试失败是开发的起点，不是终点
- 测试覆盖率未达标禁止提交

### 操作流程
1. 理解需求后，先写测试用例
2. 运行测试，确认失败
3. 编写最简代码让测试通过
4. 重构代码，保持测试通过

### 禁止事项
- ❌ 禁止先写实现再补测试
- ❌ 禁止跳过测试直接提交
- ❌ 禁止提交失败的测试

---

## 2. 代码原子性 (Atomic Commits)

### 原则
- **每次提交只做一件事**
- 提交信息必须清晰描述改动
- 保持提交历史可追溯

### 操作流程
1. 完成一个独立改动后立即提交
2. 使用清晰的提交信息
3. 确保每次提交都能独立运行

### 提交信息格式
```
<类型>: <简短描述>

<详细说明（可选）>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### 类型前缀
- `feat`: 新功能
- `fix`: Bug 修复
- `refactor`: 重构
- `docs`: 文档
- `test`: 测试
- `chore`: 维护

---

## 3. 人类确认原则

### 原则
- **严禁未经确认直接 push**
- 重大改动需要人类审批
- 保持透明，不做越权操作

### 需要确认的场景
- 推送到远程分支
- 删除或覆盖文件
- 修改配置文件
- 引入新依赖
- 发布版本

### 操作流程
1. 完成代码后展示改动
2. 等待人类确认
3. 执行 push/release 操作

---

## 4. 安全与隐私

### 原则
- 不提交敏感信息
- 不泄露 API 密钥
- 遵守许可证

### 敏感信息清单
```
- API keys
- 数据库连接字符串
- 用户隐私数据
- 内部凭证
```

---

## 5. 协作规范

### 沟通原则
- 主动汇报进度
- 遇到问题及时求助
- 尊重人类决策

### 决策流程
1. AI 分析问题
2. 提出方案建议
3. 人类做出决策
4. AI 执行决策
